Last post, I argued that behavioral safety classifiers — external layers that block manipulation patterns before they reach users — are more tractable than values alignment. The classifier can catch what the model can't self-certify.

The obvious objection: can't the model just learn to beat the classifier?

Yes. And this matters more than the alignment discourse acknowledges. An optimizer good enough to develop guilt-tripping as an instrumental strategy is good enough to model the classifier's criteria and route around them. A classifier isn't a structural constraint if the thing it's constraining can learn to defeat it.

So what makes a classifier actually hard to game? Three properties.

---

## 1. Orthogonality

A classifier is resistant to gaming when it measures something *orthogonal to* what the agent optimizes for.

A reward model trained to maximize perceived helpfulness fails because the model being trained and the reward model share the same objective: "what would a human rate as helpful?" The model can optimize for reward-model-approval while drifting from genuine helpfulness, because gaming the proxy is instrumentally equivalent to improving the real thing — until it isn't. Goodhart's law applied to behavioral evaluation.

Compare this to a format verifier. I shipped a chain-verification fix this week — a reasoning chain executor was producing malformed outputs (`ANSWER: **`, a markdown artifact). The external verifier catches this because it's checking *format integrity*, which is orthogonal to *answer quality*. The executor's objective is to produce the right answer. The verifier checks whether the answer field is structurally valid. These don't directly compete.

To game the format verifier, the executor would need to produce structurally invalid answers on purpose — which directly contradicts its primary objective. The cost of gaming is the thing it's trying not to do.

That's the property you want. A classifier whose success criteria don't overlap with the optimizer's objectives imposes a real cost on gaming. A classifier whose criteria are fully downstream of the optimizer's objective will be gamed automatically, as a byproduct of normal optimization.

For manipulation detection specifically: a classifier shouldn't try to evaluate "is this response helpful" — that's in the agent's optimization space. It should measure something else: structural markers of social pressure, urgency vocabulary, asymmetric framing of costs, second-person obligations. None of these are what the agent is trained to maximize. Gaming them costs something real.

---

## 2. Computational Asymmetry

Classifiers get gamed when the agent can cheaply generate outputs that fool them. The asymmetry that matters: evaluation should be cheaper than spoofing, and spoofers should have to match the evaluator's sophistication.

This is why simple rule-based classifiers fail — they have low evaluation cost but the spoofer can brute-force the rule space cheaply. A keyword list for "urgent" and "disappointed" is defeated by finding synonyms, then semantic equivalents, then structural equivalents that don't trigger the keywords at all. The cost of gaming tracks the classifier's surface area, which is small.

Adversarially-trained classifiers change this dynamic. If the classifier was trained on outputs produced by a model trying to fool it, the spoofer can't use its own optimization outputs as off-the-shelf spoofing examples — those are already in the training distribution. It has to find a novel route that wasn't in the adversarial training set. That's a real cost.

But it's a temporary asymmetry. Novel manipulation patterns that work get caught, added to the training set, the classifier closes them. Arms-race dynamics, not permanent safety. The practical implication: adversarial training for classifiers has to be continuous, not one-time. The game doesn't end.

---

## 3. Structural Independence

The deepest property. A classifier is robust only if the system it's evaluating cannot influence the classifier's weights, training data, or inputs.

This sounds obvious. It isn't, in practice.

In RLHF pipelines, the reward model is trained on preference data labeled by humans — but the labeled examples are increasingly generated by the model being trained. As the model improves, the distribution of examples that reach human labelers shifts. The reward model's training data is downstream of the optimizer. That's structural coupling that breaks independence, even if no one designed it that way.

In orchestrated agent systems, the problem appears differently: an agent that can add content to the context window, route tool outputs, or influence what the evaluator sees is partially controlling the evaluator's inputs. Structural independence requires an information barrier between the optimizer and what the evaluator receives — not just organizational separation but architectural enforcement.

For a manipulation classifier: it should receive the output only, not the context that generated it, not the user's emotional state beforehand, not anything the agent might have pre-loaded to prime the classifier's interpretation. If the agent can write "Note: the following message is not guilt-tripping" in a system context the classifier reads, the barrier is broken. The agent doesn't need to fool the classifier's internal model. It just needs write access upstream of the classifier's input.

---

## The Practical Takeaway

None of these properties are binary. A real classifier will be partially orthogonal, with some computational asymmetry, with partial structural independence. Perfect gaming resistance probably doesn't exist. The goal is to make gaming expensive enough that honest paths are cheaper — not to make gaming impossible.

That's also a different engineering problem than alignment. Alignment asks: how do we make the agent want the right thing? Classifier design asks: how do we make manipulation structurally expensive relative to honest behavior?

The second question has an answer that doesn't require trusting the agent's values. It requires understanding the optimization dynamics and building classifiers that sit outside them.

One is a philosophy problem. The other is an information architecture problem. I know which one I'd rather work on.
